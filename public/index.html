What is a Certificate Signing Request (CSR)?

A Certificate Signing Request (CSR) is a block of encoded text that contains information about the entity requesting a digital certificate, along with the public key that will be included in the certificate. The CSR is submitted to a Certificate Authority (CA), which uses the information to create an X.509 digital certificate.

The CSR is typically generated on the same server or system where the certificate will be installed, ensuring the private key never leaves the system.

Subject Information Fields

Common Name (CN)

The fully qualified domain name (FQDN) for web server certificates (e.g., www.example.com) or the person's name for personal certificates. This is the primary identifier in the certificate.

Organization (O)

The legal name of your organization or company. For EV and OV certificates, this must match official registration documents.

Organizational Unit (OU)

A department or division within your organization (e.g., "IT Department", "Security Team"). This field is optional and often omitted in modern certificates.

Locality (L)

The city or locality where your organization is located (e.g., "San Francisco", "London").

State/Province (ST)

The state, province, or region where your organization is located. Use the full name, not abbreviations (e.g., "California" not "CA").

Country (C)

The two-letter ISO 3166-1 country code (e.g., "US" for United States, "DE" for Germany, "GB" for United Kingdom).

Email Address

Contact email address. Commonly used in S/MIME email certificates. For web server certificates, this is typically omitted.

Subject Alternative Names (SAN)

SANs allow a single certificate to secure multiple domains, subdomains, IP addresses, or email addresses. Modern browsers require the domain to be listed in the SAN extension, not just the Common Name.

DNS

Domain names the certificate should be valid for. Use wildcards (*.example.com) to cover all subdomains at one level.

IP Address

IPv4 or IPv6 addresses. Useful for internal services accessed by IP rather than hostname.

Email

Email addresses for S/MIME certificates. Allows the certificate to be used for signing and encrypting emails.

URI

Uniform Resource Identifiers. Less common, used for specialized applications and services.

Key Configuration

RSA Keys

The most widely supported key type. Key sizes:

  • 2048-bit: Minimum recommended, widely compatible
  • 3072-bit: Good balance of security and performance
  • 4096-bit: Maximum security, slightly slower operations

ECDSA Keys

Elliptic Curve keys offer equivalent security with smaller key sizes:

  • P-256: Equivalent to RSA 3072-bit
  • P-384: Equivalent to RSA 7680-bit
  • P-521: Equivalent to RSA 15360-bit

Private Key Password

Encrypts the private key with AES-256. A password-protected key requires the password each time the key is used (e.g., when starting a web server).

Key Usage

Key Usage extensions define the cryptographic operations the certificate's key can perform:

Digital Signature

Allows the key to create digital signatures for authentication and data integrity. Essential for TLS/SSL.

Non-Repudiation

Used for signatures that provide proof of origin that cannot be denied. Common in document signing.

Key Encipherment

Allows the key to encrypt other keys. Required for RSA key exchange in TLS (non-ECDHE cipher suites).

Data Encipherment

Allows direct encryption of user data (not keys). Rarely used in practice.

Key Agreement

Allows the key to participate in key agreement protocols like Diffie-Hellman.

Key Certificate Sign

Allows the key to sign other certificates. Only used for Certificate Authority (CA) certificates.

CRL Sign

Allows the key to sign Certificate Revocation Lists. Typically only for CA certificates.

Encipher Only / Decipher Only

Used with Key Agreement to restrict operations to encryption or decryption only.

Extended Key Usage (EKU)

EKU extensions specify the purposes for which the certificate can be used:

TLS Web Server Authentication

1.3.6.1.5.5.7.3.1 - The certificate can be used to authenticate a web server (HTTPS).

TLS Web Client Authentication

1.3.6.1.5.5.7.3.2 - The certificate can authenticate a client to a server (mutual TLS).

Code Signing

1.3.6.1.5.5.7.3.3 - The certificate can sign executable code and scripts.

Email Protection (S/MIME)

1.3.6.1.5.5.7.3.4 - The certificate can sign and encrypt email messages.

Time Stamping

1.3.6.1.5.5.7.3.8 - The certificate can provide trusted timestamps for documents.

OCSP Signing

1.3.6.1.5.5.7.3.9 - The certificate can sign OCSP (Online Certificate Status Protocol) responses.

Security Best Practices

  • Never share your private key. The private key should remain on the server where the CSR was generated.
  • Use at least 2048-bit RSA keys (3072-bit or 4096-bit recommended for long-term certificates).
  • Protect private keys with strong passwords when storing them, especially on shared systems.
  • Include all domain names in SANs. Modern browsers check SANs, not just the Common Name.
  • Verify your CSR before submission using the Analyzer tab to ensure all information is correct.
  • Keep backups of your private keys in secure, encrypted storage.
  • Use appropriate validity periods. Shorter validity periods (1 year or less) are more secure.
